# Forked Fates — Product Issues, Questions, and Required Fixes

**Purpose:** Living tracker for all issues, questions, product misunderstandings, and required changes discovered during manual testing.

**Current status:** Phase 8 is technically complete, but the product is not ready for final submission until the critical product gap and UX issues below are addressed.

---

## 1. Critical Product Gap — Real LLM Agents Are Missing

### User expectation
The final product was supposed to use a real language model behind the scenes so that each character could:

- understand its own goals, traits, memories, beliefs, trust, and current situation;
- reason about the current world state;
- produce a new decision based on the changed circumstances;
- react meaningfully after an intervention;
- explain why it chose that action.

The intention was **not** to create a large collection of hardcoded `if/else` rules for character behavior.

### Current implementation
The current Live Simulation uses a deterministic, scenario-specific, rule-based provider.

- No GPT, Claude, Gemini, local LLM, or other external model is currently called.
- The LLM provider adapter exists architecturally, but no real API transport is implemented.
- Recorded mode replays stored data.
- Live mode executes the world engine, but character decisions are still generated by deterministic policies.

### Required fix
Add a real LLM-based decision mode while preserving deterministic mode for:

- tests;
- reproducibility;
- offline fallback;
- debugging;
- competition demo backup.

### Required product modes
The UI should clearly distinguish:

1. **AI Live Simulation**
   - real model calls;
   - dynamic character reasoning;
   - validated structured decisions.

2. **Deterministic Simulation**
   - rule-based;
   - repeatable;
   - mainly for testing and fallback.

3. **Recorded Demo**
   - precomputed, fixed playback;
   - no new decision generation.

### Safety and architecture requirements
- API keys must not be exposed in browser code.
- Use a small backend/proxy or secure local server.
- Each agent receives only its owned state.
- The model must return validated structured JSON.
- Invalid output must retry or fail clearly.
- Never silently fall back from AI mode to deterministic mode.
- World Engine remains the sole authoritative resolver.

**Priority: P0 — mandatory before final submission.**

## 2. Start Screen — Live vs Recorded Is Not Clear

### Current labels
- `Begin The Last Antidote`
- `Watch Recorded Demonstration`

### User confusion
Both buttons first open the same scenario briefing screen, so they appear to do the same thing.

`Watch Recorded Demonstration` also sounds like a video, while it is actually an interactive playback of precomputed data.

### Required fix
Use clearer labels and descriptions.

Suggested wording:

- **Start AI Live Simulation**
  - “Run the real agent-based simulation, fork the timeline, and compare outcomes.”

- **Explore Recorded Demo**
  - “Open a fixed precomputed run for reliable playback and inspection.”

The briefing page must display the selected mode prominently.

**Priority: P1.**

## 3. Workspace Is Too Dense and Can Confuse New Users

### Observation
The workspace shows many concepts at once:

- world view;
- timeline;
- inspector;
- controls;
- status;
- memories;
- beliefs;
- trust;
- events;
- causal information.

Some density is unavoidable, but the initial experience currently exposes too much information.

### Required fix
Use progressive disclosure.

Initially emphasize only:

- world and characters;
- current turn;
- patient state;
- Step / Run / Pause;
- latest important event;
- clear next action.

Show detailed information only after the user selects:

- a character;
- an event;
- a relationship;
- a belief;
- a memory.

Avoid showing raw identifiers or technical domain details in the default user journey.

**Priority: P1.**

## 4. Critical UX Bug — Whole Page Scrolls During Each Turn

### Current behavior
When a new turn executes, the whole page scrolls downward.

Consequences:

- world view disappears;
- controls disappear;
- the user loses context;
- the user must repeatedly scroll to read text or click controls;
- the beginning of the experience is especially confusing.

### Required behavior
The main desktop workspace should remain stable within the viewport.

- The entire page must not scroll automatically when a turn executes.
- World view and primary controls should remain visible.
- Timeline should have its own internal scroll.
- Inspector should have its own internal scroll.
- New events may scroll only inside the Timeline panel.
- New events must not steal page focus.
- Selecting or expanding an event must not move the whole page.
- Playback must preserve the user’s current scroll and inspection context.

### Recommended addition
Add:

- `Follow live events: On / Off`

When On:
- Timeline follows the latest event inside its own panel.

When Off:
- User can inspect an older turn without being pulled away.

Check for and remove or constrain:

- `scrollIntoView`;
- automatic focus calls;
- layout growth that changes document scroll;
- uncontrolled expanding sections.

**Priority: P0 UX.**

## 5. Fork Eligibility Is Hidden

### Current behavior
Some turns show a Fork option and some do not.

The user had to select turns one by one to discover which turn allowed forking.

### Required fix
Fork eligibility must be visible directly on the timeline.

Examples:

- fork icon on eligible turns;
- `Fork available`;
- disabled fork action with an explanation;
- tooltip or small label for ineligible turns.

Examples of explanations:

- `Fork unavailable: terminal turn`
- `Fork unavailable: insufficient future turns`
- `Fork unavailable: Alternate already exists`
- `Fork unavailable: only Original can be forked`

The user must never have to guess by clicking every turn.

**Priority: P1.**

## 6. Intervention Categories Need Better Explanation

### Current categories
- Information
- Item Transfer
- Environmental Event

### User question
Why are there only three options, and are they predetermined?

### Clarification
These are three **intervention categories**, not necessarily only three individual interventions.

They are currently constrained by the world schema and legal rules:

- Information: disclose a supported proposition to a character.
- Item Transfer: legally move an existing item between valid participants.
- Environmental Event: apply a supported change to a location.

Some selections are derived from the selected turn, but supported propositions, items, and environmental transitions are currently scenario-specific.

### Relationship to real LLM agents
Adding an LLM should mainly change how characters react after an intervention.

The LLM should not bypass world legality.

A future natural-language intervention could be converted into a structured event, but World Engine must still validate it.

### Required UX fix
For each category, show:

- plain-language explanation;
- what can change;
- why some options are disabled;
- which selections are based on current turn state;
- what the character may do after receiving the intervention.

**Priority: P1.**

## 7. `Use Approved Demo Intervention` Is Confusing

### Current behavior
A second button appears beside the normal Alternate creation flow:

- `Use approved demo intervention`

The user clicked it at Turn 7 and saw no visible result.

### Current purpose
This is an internal competition-demo shortcut for loading a pretested intervention configuration.

It is not a normal user feature.

It may only be valid for its configured fork turn, reportedly Turn 0.

### Problems
- “Approved” has no meaning to a normal user.
- It is unclear who approved it.
- It looks clickable even when it cannot be used.
- It gives insufficient feedback.
- It exposes internal demo/testing functionality in the public product UI.

### Required fix
Preferred solution:

- Hide it from normal product mode.
- Expose it only behind a developer/demo flag.

Alternative:

- Rename it to `Load competition demo setup`.
- Clearly state the required fork turn.
- Disable it visibly when unavailable.
- Show a reason such as:
  - `Available only when forking from Turn 0`.

A click must always produce visible feedback.

**Priority: P1.**

## 8. Twelve-Turn Model — Current Understanding

### Definition
One turn is one complete world decision round.

During each turn:

1. A frozen current world boundary is selected.
2. All four characters receive their own state.
3. Each character produces one intent.
4. The four intents are collected.
5. World Engine resolves them.
6. Events, memories, beliefs, trust, inventory, patient state, and outcome are updated.
7. A new completed boundary is created.

### Turn 0
Turn 0 is the initial world state, before any character decision round.

### Turns 1–12
These are the twelve decision rounds.

Potential maximum:

- 12 rounds;
- 4 character intents per round;
- up to 48 character decisions, including wait or failed actions.

### Fork meaning
Forking from a completed turn copies the shared history up to that boundary, applies an intervention only to Alternate, and continues future turns separately.

### Product question still open
When real LLM agents are added, decide whether:

- 12 turns remains fixed for The Last Antidote;
- the simulation may finish earlier after a terminal outcome;
- turn duration and pacing should be changed for a clearer demo.

**Priority: clarification during LLM integration.**

## 13. Alternate Appears to Copy Original — Late Interventions Produce No Visible Divergence

### User observation
The user tested Alternate twice:

- once from Turn 7;
- once from Turn 10.

After completing the Alternate and opening comparison:

- character actions appeared unchanged;
- decision text appeared unchanged;
- final outcomes appeared unchanged;
- the Alternate looked like a copied Original.

### Independent reproduction
The behavior was reproduced with Turn 7 and Turn 10 interventions.

The Alternate is technically created as a separate validated branch, so this is not simply the UI pointing to the Original object.

However, the current deterministic policies often produce the same visible action and rationale after a late intervention.

Example reproduced at Turn 7:

- Original Mara action: `Wait`
- Alternate Mara action: `Wait`
- Original rationale and Alternate rationale: identical
- Only `citedMemoryIds` changed

The Comparison module currently counts this as a “changed intent” because the cited memory changed, even when:

- action is unchanged;
- target is unchanged;
- rationale is unchanged;
- served goal is unchanged.

The UI displays action and rationale but not the differing cited memory. Therefore it shows two identical texts under “changed decisions,” which makes the Alternate look copied and makes the comparison misleading.

### Final outcome behavior
For reproduced Turn 7 and Turn 10 information/environment interventions:

- Medical remained `Lost`;
- Truth remained `Exposed`;
- Social remained `Fractured`;
- Antidote remained with Orin;
- final trust values remained unchanged.

Some hidden belief or memory state changed, but the intervention did not create a meaningful visible story divergence.

### Root causes
1. Late forks leave only a small number of future turns.
2. Scenario-specific deterministic policies may ignore or weakly react to the new information.
3. Comparison treats a cited-memory-only change as a changed decision.
4. UI does not explain whether a difference is:
   - action-level;
   - rationale-level;
   - evidence/memory-level;
   - state-only;
   - outcome-level.
5. The product permits interventions that are valid but narratively ineffective.
6. The user is not warned when an intervention created no meaningful divergence.

### Required fixes

#### A. Real LLM agents
The real LLM integration must allow characters to reason about the intervention rather than selecting the same hardcoded policy path.

#### B. Comparison semantics
Split intent differences into categories:

- `action changed`;
- `target changed`;
- `rationale changed`;
- `goal changed`;
- `evidence/cited memory changed only`;
- `no meaningful decision change`.

Do not label a cited-memory-only change as a changed autonomous decision without qualification.

#### C. Comparison UI
When action and rationale are identical but cited memories differ, show:

- `Same action, different evidence considered`

and display the relevant memory difference.

Do not show two identical decision cards as if they represent a visible decision divergence.

#### D. Ineffective intervention feedback
After Alternate resolution, calculate a meaningful-divergence score.

If there is no action, event, trust, inventory, treatment, or outcome difference, show:

- `This intervention changed internal context but did not alter the visible story.`
- `Try an earlier turn or a different intervention.`

#### E. Fork/intervention guidance
- Mark late turns as having limited remaining influence.
- Recommend turns with sufficient future horizon.
- Show a preview such as:
  - `High opportunity for divergence`
  - `Limited opportunity — only 2 turns remain`

#### F. Demo path
The competition demo must not use a path where the visible actions and final outcomes remain effectively unchanged.

### Priority
**P0 product credibility and demo clarity.**

## 14. Playback Controls Are Not Self-Explanatory — Step, Run, and Pause

### User observation
The difference between `Step`, `Run`, and `Pause` is not sufficiently explained.

The user understood that Step advances something, but:

- it was unclear what Run would do;
- Pause was never needed during testing;
- it was unclear when Pause becomes useful;
- the interface does not communicate the current playback state clearly enough.

### Intended behavior

#### Step
Execute exactly one complete simulation turn:

1. collect one intent from each character;
2. resolve all intents through World Engine;
3. update the world;
4. stop at the next completed turn boundary.

Suggested label:

- `Next Turn`

Suggested helper text:

- `Resolve one decision round and stop.`

#### Run
Automatically execute consecutive turns until one of these occurs:

- the user presses Pause;
- the simulation reaches Turn 12;
- a terminal outcome is reached;
- an error or model failure requires attention.

Suggested label:

- `Run to End`

or, if stopping at terminal conditions:

- `Auto Run`

Suggested helper text:

- `Continue through turns automatically until paused or completed.`

#### Pause
Stop automatic playback after the currently resolving turn safely completes.

Pause should not interrupt World Engine halfway through a turn, because all character intents for one turn must be resolved atomically.

Suggested helper text:

- `Stop after the current turn finishes.`

### Current UX problems
- Button labels assume the user already understands simulation playback terminology.
- Pause may be visible and clickable even when the simulation is not running.
- The interface does not clearly indicate:
  - `Ready`;
  - `Resolving Turn 4`;
  - `Auto-running`;
  - `Paused after Turn 4`;
  - `Completed`.
- The user does not know whether Run executes one turn, all remaining turns, or only animates the current events.
- The user does not know whether Pause is immediate or waits for a safe turn boundary.

### Required fixes

1. Rename or supplement the controls:
   - `Step` → `Next Turn`
   - `Run` → `Run to End` or `Auto Run`
   - `Pause` remains `Pause`

2. Add concise tooltips or helper text:
   - Next Turn: `Resolve one turn and stop.`
   - Run to End: `Continue automatically until paused or completed.`
   - Pause: `Stop after the current turn finishes.`

3. Use state-aware controls:
   - Pause disabled when not auto-running.
   - Run disabled while a turn is currently resolving.
   - Step disabled while auto-running or resolving.
   - At completion, replace or disable playback controls appropriately.

4. Show a visible playback status near the buttons:
   - `Ready at Turn 3`
   - `Resolving Turn 4…`
   - `Auto-running`
   - `Paused after Turn 4`
   - `Simulation complete`

5. On first use, show a lightweight one-time explanation or guided hint.

6. Preserve keyboard accessibility and expose button state through accessible labels.

### Priority
**P1 UX clarity.**

## 15. The Product Lacks a Real Visual World — Too Much Text and Too Little Scene Presence

### User observation
The user expected to see at least a simple 2D representation of:

- character faces or portraits;
- locations;
- where each character currently is;
- visible movement between locations;
- a stronger sense of the current scene.

Instead, the current experience is dominated by:

- text;
- boxes;
- cards;
- repeated action descriptions.

The user also questioned whether a normal user would read every character action in every turn.

### Product impact
This is not only a visual-polish issue.

Forked Fates is presented as a causal narrative simulation. Without a visible scene and recognizable characters, it feels more like:

- a debugging dashboard;
- a simulation inspector;
- a technical prototype;

rather than a living narrative world.

This reduces:

- emotional attachment to the characters;
- understanding of where events occur;
- interest in watching consecutive turns;
- perceived difference between Original and Alternate;
- demo impact.

### Required visual direction
The product does not need a full game engine or complex animation.

It needs a lightweight but clear 2D narrative layer.

#### Character representation
Each character should have:

- a distinct portrait or illustrated avatar;
- name and role;
- recognizable visual identity;
- current emotional/status indicator;
- current location.

#### Location representation
Show the three locations as a simple 2D scene or map.

Each location should have:

- a clear visual identity;
- a label;
- characters currently present;
- relevant object or event indicators;
- visible state changes when applicable.

#### Movement
When a character changes location:

- animate or visually trace movement;
- update the scene without scrolling the page;
- make the movement understandable at a glance.

#### Current scene focus
The active location or important event should be visually emphasized.

The user should be able to understand the current world state without reading all raw action cards.

### Text hierarchy problem
The current UI asks the user to read too much.

A user should not need to read all four full action descriptions on every turn.

### Required information hierarchy

#### Default turn summary
Show one short narrative summary, for example:

- `Mara remains with Niko while Dain confronts Orin near the storehouse.`

#### Character actions
Show compact character chips or one-line summaries:

- `Mara — stayed with Niko`
- `Dain — questioned Orin`
- `Sera — moved to the clinic`
- `Orin — concealed the key`

#### Detailed reasoning
Only show full:

- rationale;
- memories;
- beliefs;
- trust;
- cited evidence;
- technical event details;

after the user explicitly opens a character or event.

### Recommended turn presentation
Each turn should visually prioritize:

1. What changed in the world?
2. Who moved?
3. Who interacted?
4. What important new fact or conflict appeared?
5. Did patient, antidote, trust, or outcome state change?

Raw per-agent decision details should be secondary.

### Relationship to the fixed scenario
The fixed scenario may contribute to emotional distance, but it is not the main cause.

Even a fixed scenario can feel engaging when:

- characters have visible identities;
- locations are understandable;
- actions are animated or visually summarized;
- the user can immediately see the consequences of an intervention.

### Required implementation decision
Add a visual narrative layer before final submission:

- illustrated/static 2D character portraits;
- simple 2D location map or scene;
- visible character placement;
- lightweight movement animations;
- event/state icons;
- compact turn summaries;
- collapsible detailed reasoning.

Do not replace the deterministic world engine or technical inspectors. Keep them as a secondary analysis layer.

### Priority
**P0/P1 — central to product identity and demo impact.**

## 16. Product Vision — User-Created Scenarios and Playable AI-Driven Worlds

### User vision
The intended final product is broader than the fixed `The Last Antidote` scenario.

A future user should be able to:

- write or describe their own scenario;
- define or generate characters;
- define locations, conflicts, goals, secrets, relationships, and important objects;
- let AI agents interpret the scenario;
- watch the characters act inside a visible 2D world;
- intervene during the story;
- fork the timeline;
- compare alternate outcomes;
- replay or explore different versions of the same scenario.

The product should feel like a combination of:

- an AI narrative sandbox;
- a scenario creation tool;
- an interactive story;
- a causal simulation;
- a lightweight game.

### Current scope
The current build supports only one fixed demonstration scenario:

- `The Last Antidote`

It does not currently include:

- a scenario editor;
- arbitrary user-written scenario import;
- automatic character generation;
- automatic location generation;
- general scenario compilation;
- dynamic visual scene generation.

### Demo messaging requirement
The demo must clearly separate:

#### What exists now
- one complete proof-of-concept scenario;
- causal world state;
- multiple characters with private context;
- interventions;
- timeline forks;
- Original vs Alternate comparison.

#### What the final product is intended to become
- a platform where any user can create an AI-driven scenario and watch, play, intervene, and explore alternate outcomes.

Do not claim that user-created scenarios already work.

### Recommended demo wording
Use a statement close to:

> “The Last Antidote is our first playable proof of concept. Our larger vision is a creator platform where anyone can describe a scenario, generate its characters and world, and then watch, influence, and fork an AI-driven story as it unfolds.”

### Product framing
The fixed scenario should be presented as:

- a vertical slice;
- a proof of concept;
- an example world demonstrating the underlying causal simulation architecture.

It should not be presented as the full intended product.

### Roadmap direction
After the current product is stabilized:

1. Scenario schema.
2. Scenario authoring UI.
3. LLM-based scenario compiler.
4. Character and relationship generation.
5. Location and object generation.
6. Validation of generated worlds.
7. Reusable visual scene templates.
8. User scenario library and replay.
9. Sharing and collaborative scenario exploration.

### Priority
**P1 for demo messaging; post-submission roadmap for implementation.**
